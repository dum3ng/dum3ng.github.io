In the html4 standard, we can use window.location to programmatically change the url and navigate somewhere.

html5 new web api

The html5 bring a new web api web.history,

google closure library

The google closure library has capsulated a goog.History and goog.history.Html5History object for their native web api.

goog.History

The goog.History will listen for the hashchange event, and will trigger a goog.historyEvent we can listen to. The historyEvent has a token property which represents the current history.

goog.history.Html5History

The goog.history.Html5History will listen both for popstate and hashchange event, and do the same with goog.History.

use History

Since the History object take advantage of the hash tag #, you can keep all the href property link tag <a> unmodified, as click on them does not cause a new request at all. This will make things got some easier.

The disadvantage

The url with # can be marked as bookmark and can be used as a normal url to manully put in the browser address bar, but they can not be crawed by search engines like Google. That is why Google suggest to use the new history.pushState() to replace the location.hash.

Conclusion

If we are using the same code to make a mobile app, then we can just directly use a path state to maintain current page state and keep the url untouched (despite in fact there is no visible url). But as a web app, we need connect the url with different app state, i.e. app views, thus a listener for a history navigate event will be neccesary. The history itself will listen the native web event popstate and hashchange when url changed.

The three conditions when url changed:

  • manually edit in the browser address bar
  • pushed back or forward button
  • click on a hyperlink

For the first condition, the browser will always make a new request, that is not in the consideration of the client side (that will be in the scope of routes defination at server).

For the second condition, the browser will never make a new request, and native web event will be triggered.

For the third one, there are two additional situations: if the href begins with #, the navigation will stay in current document, and can be listened; If it is a new location, you should add an event handler to prevent default behaviour since it will make a new request by default, and dispatch a customized handler.
